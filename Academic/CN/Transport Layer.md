# Transport Layer — Complete Notes

---

# 1. Introduction to Transport Layer

The **transport layer** is the **fourth layer in the OSI model** and sits between the **application layer** and the **network layer**.

Its main purpose is to provide **logical communication between application processes running on different hosts**.

Unlike the network layer which provides **host-to-host communication**, the transport layer provides **process-to-process communication**.

---

## Key Responsibilities

The transport layer performs several important functions:

1. **Process-to-process delivery**
    
2. **Segmentation and reassembly**
    
3. **Reliable data transfer**
    
4. **Flow control**
    
5. **Congestion control**
    
6. **Error detection**
    
7. **Multiplexing and demultiplexing**
    

These services ensure efficient communication between applications across networks.

---

# 2. Transport Layer Services

Transport layer protocols provide **logical end-to-end communication** between processes.

Even though packets travel through routers and networks, the transport layer creates the illusion of **direct communication between two processes**.

---

## Sender Side Operations

At the sender:

1. Receives **application message**
    
2. Breaks message into **segments**
    
3. Adds **transport header**
    
4. Sends segments to **network layer**
    

---

## Receiver Side Operations

At the receiver:

1. Receives **segments from network layer**
    
2. Checks **header values**
    
3. Reassembles **segments into message**
    
4. Delivers message to **application process**
    

---

# 3. Transport Layer vs Network Layer

|Feature|Transport Layer|Network Layer|
|---|---|---|
|Communication type|Process-to-process|Host-to-host|
|Unit|Segment|Packet/Datagram|
|Example protocols|TCP, UDP|IP|
|Reliability|May provide reliability|Best-effort|

---

Consider two houses:

# 5. Multiplexing and Demultiplexing

## Multiplexing

Occurs at the **sender side**.

Data from multiple applications are:

1. Collected
2. Encapsulated into segments
3. Sent to network layer

Each segment contains **header information** used later for delivery.

---

## Demultiplexing

Occurs at the **receiver side**.

The transport layer uses **header information** to deliver segments to the correct application socket.

Host receives IP Datagrams (Each datagram has source IP and destination IP). Each datagram contains 1 transport layer segment (each segment has source and destination port numbers). Host uses IP address and Port Numbers to direct the segment into the appropriate socket. 

---

### Example

Multiple applications:

- Browser
    
- Skype
    
- Netflix
    

When data arrives, the transport layer determines **which process should receive it** using **port numbers**.

# 8. Connectionless Demultiplexing (UDP)

When creating a socket, we must satisfy host-local port number specified. When creating a datagram to send into a UDP socket, we must specify the destination IP and Port.

In UDP:
Demultiplexing uses **only destination port number**.
Segments with:
- same destination port
- different source IPs

are delivered to the **same socket**.

# 10. UDP (User Datagram Protocol)

UDP is a **connectionless transport protocol**.

Characteristics:
- No connection establishment
- No reliability
- No ordering
- No congestion control

Thus UDP provides **best-effort delivery**.
Just hopes that the data gets sent. Hence, there's no need for a handshake.

---

## Why UDP Exists

UDP exists because it is:

1. **Fast**
2. **Low overhead**
3. **No connection delay**
4. **Useful for real-time applications**

---

## Applications Using UDP

Examples include:

- DNS
    
- SNMP
    
- Streaming media
    
- Online games
    
- HTTP/3
    

These applications can tolerate **some packet loss**.

# 11. UDP Segment Structure

UDP segment format:

```
--------------------------------
| Source Port | Destination Port |
--------------------------------
| Length | Checksum |
--------------------------------
| Data (Payload) |
--------------------------------
```

Fields:

- **Source Port** → sender application
    
- **Destination Port** → receiver application
    
- **Length** → total segment size
    
- **Checksum** → error detection
    

---

# 12. UDP Checksum

Purpose: **detect transmission errors**.

Sender:

1. Divide data into **16-bit words**
    
2. Add them using **one's complement addition**
    
3. Store result as checksum
    

Receiver:

1. Compute checksum again
    
2. Compare with received checksum
    
3. If mismatch → error detected
    

---

### Limitation

Checksum provides **weak error detection**.

Certain bit errors may still go undetected.

# 13. Reliable Data Transfer (RDT)

Reliable data transfer ensures:

- No data loss
    
- Correct order
    
- No corruption
    
- No duplication
    

However, the underlying network channel may be **unreliable**.

---

## Reliable Service Abstraction

To the application, transport layer appears as a **perfect reliable channel**.

But internally it operates over an **unreliable network layer**.

# 16. rdt1.0 — Perfect Channel

Assumptions:

- No packet loss
    
- No corruption
    

Sender:

```
rdt_send(data)
send packet
```

Receiver:

```
receive packet
deliver_data()
```

No need for ACKs.

---

# 17. rdt2.0 — Channel with Bit Errors

Now channel may **corrupt packets**.

Solution:

- Use **checksum**
    
- Use **ACK/NAK**
    

---

### ACK

Acknowledgment that packet received correctly.

---

### NAK

Negative acknowledgment indicating packet error.

Sender retransmits packet when NAK received.

---

# 18. Stop-and-Wait Protocol

Sender sends **one packet at a time** and waits for ACK before sending next packet.

Advantages:

- Simple
    
- Reliable
    

Disadvantages:

- Very inefficient
    

---

# 19. rdt2.0 Problem

If **ACK or NAK becomes corrupted**, sender cannot know receiver state.

Solution:

- Add **sequence numbers**
    
- Detect **duplicate packets**
    

---

# 20. rdt2.1

Improvements:

- Sequence numbers (0,1)
    
- Detect corrupted ACK/NAK
    
- Handle duplicates
    

Receiver keeps track of expected sequence number.

---

# 21. rdt2.2

Further improvement:

- Remove NAK
    
- Use only ACK
    

Receiver sends ACK for **last correctly received packet**.

Duplicate ACK acts like NAK.

---

# 22. rdt3.0 — Packet Loss

Now channel may:

- Lose packets
    
- Lose ACKs
    

Solution:

- Use **timeout mechanism**
    

---

### Timeout

Sender starts timer after sending packet.

If ACK not received within time:

```
retransmit packet
```

---

# 23. Stop-and-Wait Performance

Example:

```
Link speed = 1 Gbps
Packet size = 8000 bits
Propagation delay = 15 ms
```

Utilization becomes extremely low.

Thus stop-and-wait wastes bandwidth.

---

# 24. Pipelined Protocols

To improve efficiency:

Sender sends **multiple packets without waiting**.

Advantages:

- Better link utilization
    
- Higher throughput
    

Requires:

- Larger sequence numbers
    
- Buffers at sender/receiver
    

---

# 25. Go-Back-N Protocol

Sender can send up to **N packets** without ACK.

Concept:

**Sliding Window**

```
send_base → first unACKed packet
nextseqnum → next packet to send
```

---

### Go-Back-N Rules

Receiver:

- Accepts only **in-order packets**
    
- Discards out-of-order packets
    

Sender:

If timeout occurs:

```
retransmit all packets starting from lost one
```

---

# 26. Selective Repeat Protocol

Improvement over Go-Back-N.

Receiver:

- Accepts **out-of-order packets**
    
- Buffers them
    

Sender:

- Retransmits **only lost packets**
    

---

### Features

- Higher efficiency
    
- More complex implementation
    
- Requires more buffering

Transport layer provides **end-to-end communication between applications**.
