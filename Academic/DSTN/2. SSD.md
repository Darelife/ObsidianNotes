# HDD vs SSD

- HDDs have spinning magnetic platters, mechanical arm, R/W Head
- Components in SSD : Controller, NAND Flash, DRAM (No moving parts)
- No moving parts improves reliability, durability, and power efficiency
- HDDs : 200 IOPS, SSD : 100,000 IOPS
- HDD: 5–10 ms
- SSD: 50–100 µs


# Nand cells
- SSDs have a bunch of NAND cells, that also help with concurrency (we can use multiple NAND cells at the same time)
- here, we have 3 ops : read, write, and erase (hdds don't have erase)
- we have to erase an entire block, and then only can we write to its pages
- Need to use a garbage collector (overhead)
- There's a limit to the amount of times that we can erase
- We can Read/Write to pages, but we can erase only blocks (not pages)
- Blocks > Pages
- Pages : 4-16 KB
- Blocks : 128 - 256 Pages
- Can't overwrite data directly, we'll need to erase the block, and then, write in it.
- We can write in empty pages, but if we need more space, we'll have to erase an entire block, and then, write to empty pages in it.

# FTL Mapping Strats (Page | Block | Hybrid)

## Page (didn't understand) (4 KB granularity)
Each LBA gets mapped to a Page.
It's flexible, and has a low WAF, but a huge ram requirement
(1 TB SSD $\rightarrow$ 1 GB Mapping Table)

## Block Level Mapping (256 KB granularity)

Each LBA is mapped to a block, instead of a page. 
Cuts RAM usage significantly (~64x less), compared to page level.
But the WAF increases, because we'll have to erase more stuff (as we're gonna write to a large area every time)

## Hybrid
1. Block Mapping Table (LBN -> PBN)
2. Log Page Mapping Table (LPMT) ((LBN, page_offset) → (LogBlock, page))

Eg:
```
LBN 10 → PBN 100

PBN 100 (DATA BLOCK):
  Page 0 → (10,0)
  Page 1 → (10,1)
  Page 2 → (10,2)
  Page 3 → (10,3)

Step 2: Overwrite ONE page

Host overwrites logical page (10,2).

What NAND forbids

Cannot overwrite page 2 in PBN 100

Cannot erase PBN 100 (it has valid data)


PBN 300 (LOG BLOCK):
  Page 0 → updated (10,2)

update log
(10,2) → (300,0)

PBN 100 (DATA):
  Page 2 = OLD (invalid logically, but not erased)

# Step 3: Another overwrite (same block)

Host overwrites **logical page (10,0)**.

### Action

- Write into same log block
    

`PBN 300 (LOG BLOCK):   Page 0 → (10,2)   Page 1 → (10,0)`

### Update mapping

`(10,0) → (300,1)`

---

# Step 4: Reads (important!)

Read `(10,2)`:

1. Check log mapping → FOUND
    
2. Read from `PBN 300, Page 0`
    

Read `(10,1)`:

1. Not in log table
    
2. Read from `PBN 100, Page 1`

# Step 5: Log block fills up

Host overwrites `(10,1)` and `(10,3)`.

`PBN 300 (LOG BLOCK):   Page 0 → (10,2)   Page 1 → (10,0)   Page 2 → (10,1)   Page 3 → (10,3)`

Log block is **full**.

# Step 6: Merge operation (the unavoidable erase)

Now hybrid FTL **must clean up**.

### Case here: **Switch merge** (best case)

Because log block contains **ALL pages of LBN 10**.

### Actions

1. Update block mapping:
    

`LBN 10 → PBN 300`

2. Erase old data block:
    

`Erase PBN 100`

3. Log block becomes the new data block
    

🔥 **No page copying needed** — very cheap!

Assume only `(10,2)` and `(10,0)` were updated.

### Partial merge steps

1. Allocate new block `PBN 400`
    
2. Copy:
    
    - Updated pages from log block
        
    - Unchanged pages from old data block
        
3. Erase:
    
    - Old data block (100)
        
    - Log block (300)
        
4. Update:
    

`LBN 10 → PBN 400`

This is where write amplification happens.
```

# Out of Place Update: Elimination Erase before Write
- Suppose we need to edit an old page
- Write new page elsewhere instead of erasing, as we can't overwrite
- Update the mapping
- Mark the old page as invalid

Eventually, we can convert random to sequential via this.
When the SSD writes updates, it doesn't go back to the original location.
Instead it writes to the next available free page. Hence, sequential

Later on, the garbage collector sees that the old block has a lot of invalid pages. So, it will copy the valid pages to some other block, and then, erase the old block, to reuse it.

# Memory Table Recovery, & Power Loss Protection
(out of band area in nand flash blocks)

- FTL Memory is stored in DRAM. So, if the power fails, all the data is lost.
- Hence, we can store the meta data in NAND. We'll use journaling, and checkpoints.
- In journaling, the SSD will basically keep noting down the changes that are being made in the SSD, and like, it will
- In storing the meta data, for each block, we'll store the LBA, time stamp and the validity flag (How will it work if we can't overwrite?)
- Checkpoints are stored with a bunch of journal entries. Then, for recovery, it loads the journal entries, after the last checkpoint.
- Enterprise SSDs include capacitors. These capacitors store enough energy to keep the controller running for a few milliseconds after power loss. So, in those few milliseconds, the controller flushes DRAM mapping to NAND, commits the journal, and finishes the in flight writes.

# Garbage Collection

- Select victim block
- Copy valid pages
- Ease block

Efficiency depends on the number of invalid pages. More is good, as we won't have to copy them somewhere else, in order to erase the old block.


## GC Policies

### Greedy (Choose the block with the most Invalid Pages)
- Simple
- But uneven wear

### Cost Benefit Policy
- Considers the number of invalid pages, and the age of the block (number of erases)
- More balanced

### CAT Policy
- Predicts when the pages die together, and groups them to reduce fragmentation.

# Over Provisioning

# Wear Leveling

Flash blocks wear out.
Wear leveling distributes writes evenly.

## Dynamic wear leveling

Moves new writes to least worn blocks.

## Static wear leveling

Also moves cold data from old blocks.
Ensures all blocks age evenly. Then, use the old block, that still has a lot of life left, and then, until it wears a lot, keep using it, while giving the cold data (new block) some rest, as it's less likely to be modified.

# Data Retention

Flash cells lose charge over time.
Mechanisms:
- tunneling leakage
- defects
- thermal emission.

## Temperature effect

Higher temperature → faster charge loss.

Rule:
Every 5–10°C decrease doubles retention time.

## Disturb errors

Read disturb
Program disturb

## Mitigation

- ECC
- read scrubbing
- data refresh.
